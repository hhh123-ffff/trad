# 个人量化交易平台系统架构（第一阶段）

## 1. 产品范围

本项目定位为“个人量化交易平台”，包含三种运行模式：

1. 研究模式：因子计算与历史回测。
2. 模拟盘模式：定时生成信号并进行虚拟持仓更新。
3. 监控模式：资产看板、风险状态与系统运行日志。

V1 版本不直连真实券商账户，但会预留统一 Broker 适配接口，便于后续升级到实盘。

## 2. 总体架构

系统采用“先模块化单体、后服务拆分”的工程路线，保证 0 到 1 快速可落地，同时具备扩展能力。

- 前端层（Next.js + TypeScript）：负责可视化页面与交互。
- API 层（FastAPI）：提供统一 REST 接口，编排量化各子模块。
- 量化引擎层（Python）：数据、因子、策略、回测、模拟盘、风控。
- 调度层（APScheduler）：每日定时执行数据更新、信号生成、风控巡检。
- 存储层（PostgreSQL/SQLite）：保存行情、因子、信号、订单、持仓、资产、风险日志、回测结果。
- 日志层（loguru）：记录任务执行链路与异常信息。

## 3. 模块职责

### 3.1 数据模块
- 从 `akshare`（默认）或 `tushare`（备用）采集 A 股日线行情。
- 进行字段标准化与数据清洗。
- 通过 upsert 写入 `prices`，唯一键 `(stock_id, trade_date)`。
- 股票基础信息写入 `stocks`。

### 3.2 因子模块
- 动量因子：20 日、60 日收益率。
- 成交量因子：成交量滚动比值或 z-score。
- 波动率因子：20 日年化波动率。
- 基本面因子：预留输入接口，后续接财务数据。
- 结果写入 `factors`。

### 3.3 策略模块
- 多因子加权评分模型。
- Top N 选股。
- 每周调仓。
- 择时过滤：沪深 300 均线规则。
- 输出写入 `signals`。

### 3.4 回测模块
- 历史区间回测。
- 输出指标：年化收益、最大回撤、夏普比率、胜率、总收益。
- 回测概览写入 `backtest_runs`，净值曲线写入 `backtest_nav`。

### 3.5 模拟交易模块
- 将信号转换为模拟订单。
- 更新持仓与账户资产快照。
- 交易记录写入 `orders`，持仓写入 `positions`，资产净值写入 `portfolio`。

### 3.6 风控模块
- 单票仓位限制。
- 固定止损规则（-8%）。
- 组合最大回撤告警。
- 总仓位暴露控制。
- 告警写入 `risk_logs`。

### 3.7 API 模块
- `/dashboard`：资产总览 + 净值曲线。
- `/positions`：当前持仓与浮盈浮亏。
- `/orders`：模拟交易记录。
- `/signals`：策略信号。
- `/backtest`：回测结果。
- `/risk`：风险状态与告警信息。

### 3.8 前端模块
- Dashboard 页面。
- 持仓页面。
- 交易记录页面。
- 策略信号页面。
- 回测页面。
- 风控页面。

## 4. 每日运行流程

1. 调度器在收盘后触发 `update_market_data`。
2. 数据模块采集并入库。
3. 因子模块更新最新因子。
4. 策略模块生成下一交易日信号。
5. 模拟盘模块更新订单、持仓与账户净值。
6. 风控模块执行规则并记录风险事件。
7. 前端通过 API 拉取最新数据进行展示。

## 5. 可扩展设计

- Broker 适配器目录：`backend/app/services/brokers/`
  - `get_account()`
  - `place_order()`
  - `cancel_order()`
  - `get_positions()`
- 数据源适配层：支持 akshare/tushare 切换。
- 策略接口层：支持多策略并行。
- 风控规则注册机制：支持规则插件化。

## 6. 稳定性与可运维基线

- 使用数据库约束与索引保证数据一致性。
- 调度任务幂等化（基于 upsert）。
- 结构化日志记录任务 ID 与执行耗时。
- 提供健康检查接口，支持容器健康探针。
- 使用 Docker Compose 保证本地部署可复现。

## 7. 依赖稳定性说明

- `akshare` 受上游数据站点影响，字段偶尔波动。
- 备用方案：切换到 `tushare` 数据适配器。
- 回测引擎优先 `vectorbt`（研究效率高）。
